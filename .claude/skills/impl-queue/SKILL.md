# 実装キューの消化

`docs/plans/README.md` の番号つきキューが実装順の正。各計画ファイル冒頭の「状態」が進捗の正。
未着手の計画を上から順に、直列で実装する。

## 1計画ごとのサイクル（完結させてから次へ）

1. `docs/plans/README.md` を読み、最初の「未着手」の計画を特定する
2. 計画ファイルを読む。引き継ぎメモに途中状態があればそこから再開する
3. 実装する。受け入れ条件は AGENTS.md の規約どおり可能な限り自動テストとして残す
4. ビルドと動作確認、受け入れ条件の確認（`npm test` / `npm run validate:content` を通す）
5. Breaker レビューを行う（AGENTS.md 参照。仕様準拠でなく「壊れるケースの発見」）
6. 計画ファイル冒頭の状態を「完了（日付）」に更新し、引き継ぎメモに実装の要点
   （実装場所・設計判断・残った注意点）を書く。次のセッションが文脈なしで引き継げる粒度にする
7. main にコミット＆プッシュ。CI（check → build → deploy）が通ることを確認してから次の計画へ

## 守ること

- **コミット前に `git status` を確認し、自分の変更だけをコミットする。**
  対象が明確なら `git commit -- <パス>` でパスを限定する。
  理由: 作業ツリーを複数セッションが共有することがあり、`git add` 任せのコミットは
  他セッションのステージ済み変更を巻き込む（実際に過去へ巻き込み事故があった）
- **各計画の「非スコープ」には手を出さない。** 非スコープには「どの条件で破綻するか」が
  書いてあり、それは意図的な先送りの記録。計画に書かれていない判断が必要になったら、
  自分で進めずにユーザーに確認する
- **利用制限の一時停止ルール（CLAUDE.md）に従う。** 停止するときは途中状態を該当計画の
  引き継ぎメモに書いてから止める（通知と自動再開スケジュールの登録も CLAUDE.md のとおり）
- 番号つきキューを終えたら、README の「並行トラック」の未着手計画も同じサイクルで続ける。
  前提条件（「◯◯の完了後」等）が README に書かれている場合はそれに従う
- キューを最後まで終えるか、停止条件に達するまで続ける
