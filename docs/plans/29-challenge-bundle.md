# 今日の課題の挑戦束（楽・ふつう・挑戦の自己選択）

- 状態: 未着手
- 触るファイル範囲: `src/lib/goals.ts`（束の生成）・`src/screens/HomeScreen.tsx`（今日の課題カード）
- 出典: ゲーム調査 A-3（挑戦選好: 6割が挑戦的目標を選好）・§0（自己選択×適正挑戦が失敗の2大要因を塞ぐ）

## 目的

今日の課題を1本の押し付けにせず、難度の異なる束から本人に選ばせる。
自律性（自分で選ぶ）と適正挑戦（有能感）を1つのUIで満たす。

## 内容

- 今日の課題に**「ふつう」と「挑戦」の2束を並置**し、本人が選ぶ。どちらを完遂しても週目標（28）への寄与は同等
- **挑戦束の対価は「量の圧縮」**: ふつう束10問に対し挑戦束は難問4問など、少数で同じ寄与
  （XP倍率ボーナスは導入しない。外発報酬を厚くするのではなく時間効率を報酬にする = アンダーマイニング回避）
- 挑戦束の中身:
  - 数学: difficulty=3（応用）中心
  - 数学以外（難易度タグ未整備の教科）: **input 形式中心**で難度を作る（choice より上の段。タグ整備を待たない）
- **楽な方を選んでも一切責めない**: 文言・演出・達成祝福はどちらも同等。「挑戦しない日があってよい」前提を仕様として固定

## 非スコープと破綻条件

- 束のカスタム編集はやらない → 壊れた契約（28の自由入力禁止と同じ理由）
- 3束以上の細分化はやらない → 選択肢過多で選ぶこと自体が負担になる
- 該当教科に input/応用問題が足りない日は挑戦束を出さない（無理に水増しして「挑戦なのに簡単」を作らない）

## 効果の見方

- 挑戦束の選択率（選ばれなければ難度・量の調整シグナル）
- 今日の課題の完遂率の前後比較

## 受け入れ条件

- 今日の課題に2束が並び、選択した束で課題が進む
- どちらの束でも週進捗への寄与が同等
- 挑戦束の構成（数学=応用中心・他教科=input中心）がテストで確認できる
- 既存の週間目標・課題再計算（28）の動作に影響がない
