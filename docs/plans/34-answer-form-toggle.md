# choice/input の手動切替（出題形式の本人コントロール）

- 状態: 完了（2026-06-12）
- 引き継ぎメモ:
  - 切替UI: QuizScreen の `formOverride` state（毎問 null = エンジンのデフォルトに戻る）。
    対象は「answers つき choice」のみ（input→choice はスキーマ上選択肢が無いため構造的に不可）。
    写経・order・flashcard・模擬テストは対象外
  - **証拠ルールの解釈（オーナー確認推奨）**: 計画の「input 段への昇格は input で解いた正解のみが証拠」を
    「現在の段が要求する形式より**下**の形式で解いた正解は、その段の昇格の連続に数えない（降格もしない）」と
    実装した（mastery.ts applyAnswer の form 判定）。choice 段での choice 正解による従来の自動昇格
    （0→1）は維持＝切り替えない利用者の進行を壊さない。spec.md に不変条件を追記済み
  - エンジンがフォールバックで下の形式を提示した場合（input 変種なしの concept 等）も同じルールが
    適用される＝その concept は上の段の証拠を作れず実態どおりに止まる（意図どおり）
  - form 未指定の経路（模擬テスト・レッスン）は従来動作（後方互換）
  - 履歴: `QuestionDayStat.switchUp / switchDown`（エンジン推奨から上げた/下げた解答数）。
    フィクスチャ V1_AFTER_28 を追加（計画29-34 のフィールドが無い旧形式の互換テスト）
- 触るファイル範囲: `src/screens/QuizScreen.tsx`（切替UI）・`src/lib/mastery.ts`（解答形式ベースの昇降格）・`src/types.ts`（履歴に解答形式）・`docs/spec.md`（不変条件の追記）
- 出典: オーナー決定。12の適応出題（段に応じた形式の自動決定）への上書き手段
  （AGENTS.md プロダクト設計原則: 推奨はデフォルト動作・本人の上書きが常に勝つ、の具体化）

## 目的

出題形式（choice/input）をエンジン任せにせず、本人が問題ごとにいつでも選び直せるようにする。
エンジンの推す形式はデフォルトとして残す。

## 内容

- 問題画面に常設の切替（「選択肢で解く ⇄ 自分で入力する」）。デフォルトは習熟度エンジンの推す形式。
  **解答確定前ならいつでも切替可能**
- **習熟度の証拠は「実際に解答した形式」で数える**:
  - 解答履歴に解答形式を記録する（フィクスチャ追加・後方互換の約束に従う）
  - input 段への昇格は「input で解いた正解」のみが証拠
  - input 推奨の問題を choice に切り替えて正解 → choice 段の証拠として数える（それだけでは降格しない）
  - 「いつも choice に下げる」使い方をしても習熟度が実態どおりに止まるだけで、ズルにも罰にもならない
- **切替を出すのは両形式をレンダリングできる問題だけ**（choice→input は常に可。input→choice は
  選択肢が定義されている場合のみ）。出せない問題ではボタン自体を出さない（07ヒントと同じ方針）。
  誤答肢の整備は計画14/23のコンテンツ側課題として送る
- XPは形式によらず現行どおり（1発正解 +10）。変更しない
- 実装時に spec.md へ追記: 「昇降格の証拠は提示形式ではなく解答形式ベース」

## 非スコープと破綻条件

- 切替の恒久記憶（常に input で出す等の設定）はやらない → 毎問エンジンのデフォルトに戻る。
  31のヒント2択と同じく毎回の能動的選択に意味がある。煩わしいという声が出たら記憶式を検討
- order・写経・フラッシュカード形式は切替対象外（choice⇔input のペアのみ）
- 模擬テストでは切替を出さない（「本番に近い形＝choiceのまま」という12の判断を維持）

## 効果の見方

- 切替の利用率と方向（上げる切替/下げる切替の比率）。挑戦束（29）内での切替は29の選択率と合わせて観察

## 受け入れ条件

- 切替で同じ問題が choice/input 両形式で解け、進捗キーが同一であること（既存の不変条件を維持）
- 解答形式が履歴に記録され、昇格判定が解答形式ベースで動くことがテストで確認できる
- 選択肢のない問題・模擬テストで切替が出ない
- 既存データ互換（フィクスチャ追加）
