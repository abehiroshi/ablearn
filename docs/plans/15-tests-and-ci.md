# 自動テストとCIゲート（回帰防止の基盤）

- 状態: 完了（2026-06-12）
- 触るファイル範囲: `src/lib/` のテスト追加・`docs/spec.md`（新規）・`.github/workflows/`・`scripts/`・`package.json`
- 引き継ぎメモ: Vitest 2.x・`npm test`（35テスト）。互換フィクスチャは `src/lib/__fixtures__/appstate.ts`（スキーマ変更時はここに変更前形式を追加）。コンテンツ検証は既存 `scripts/validate-content.mjs` を lesson/新フィールド対応に拡張し `npm run validate:content` 化。CI は check ジョブ（tsc＋test＋validate:content）→ build → deploy の直列。spec.md への昇降格ルール追記は12の実装時

## 目的

機能追加時の回帰を防ぐ。仕様の主体は自動テスト（計画の受け入れ条件をテストとして残す）に置き、
文書はコードから読み取りにくい不変条件だけに絞る（AGENTS.md「ドキュメントを減らす方向」と整合）。
11〜12の中核変更より先に入れる。

## 内容

### テスト（Vitest・`npm test`）

- 対象は `src/lib/` の純粋関数: 正規化・正誤判定（quiz.ts）、streak・履歴集計（storage.ts）、
  モード判定・おすすめ優先度（recommend.ts）、統計（stats.ts）
- **localStorage 後方互換テスト（最重要）**: 過去の各時点のデータ形式のフィクスチャ
  （v1初期 / 04の履歴追加後 など）を読み込んで壊れないことを常時検証する。
  以後スキーマを変えるたびに、変更前の形式をフィクスチャとして追加する
- UIコンポーネントテストは当面やらない（下記の非スコープ）

### 仕様書（`docs/spec.md`・1ファイル・薄く）

- 不変条件だけを書く: XPの額（+10/+5/+2/フラッシュカード+5）、localStorage 後方互換の約束
  （既存キーを壊さない・問題idを変えない）、モードの導出定義、模擬テストのルールなど
- 画面の見た目・操作手順は書かない（変わるたびに腐るため）
- 12実装時に昇降格ルールを追記する

### CIゲート（GitHub Actions）

- 現在の「push → 即デプロイ」を「push → 型チェック＋テスト＋コンテンツ検証 → 通ったらデプロイ」に変更
- コンテンツ検証はここで簡易版を先行導入（スキーマ準拠・セット内id一意。[14](14-content-pipeline.md) で拡張）
- main 直 push 運用のまま「壊れたコードが本番に届かない」を保証する安全装置

### 運用ルール

- AGENTS.md に1行追加: 実装時、計画の受け入れ条件は可能な限り自動テストとして残す

## 非スコープと破綻条件

- UIコンポーネントテスト・E2E はやらない → レイアウト崩れ・操作感の回帰は検出できない。
  実機確認で拾う。頻発するならスモークテストを検討
- カバレッジ目標は置かない → 数字合わせのテストを書く誘因になるため

## 受け入れ条件

- `npm test` がローカルと CI の両方で通る
- 旧形式フィクスチャの互換テストが存在し、わざとスキーマを壊すと落ちる
- テストが失敗する push はデプロイされない（GitHub Actions で確認）
- `docs/spec.md` が存在し、上記の既存不変条件を網羅している
