# コンテンツ生成手順書（概念ファミリー）

AIセッションでコンテンツを生成するときのプロンプトテンプレートと手順。
このファイルだけを読めば、セッションの文脈なしで同じ品質で生成できることを目的とする。
スキーマの正は [src/types.ts](../src/types.ts)、規約の正は [README](../README.md)（コンテンツの追加・更新）と
[docs/spec.md](spec.md)。学習設計の根拠は
[調査レポート](research/learning-apps-survey.md) C-1（worked-example 効果）・C-2（解法プロセスの明示的指導）。

## 入力（依頼者が渡すもの）

```
教科: （例: 理科）
単元: （例: 化学変化と原子・分子）
範囲: （教科書の該当箇所・扱う概念のメモ。例: 化学反応式の係数合わせまで）
links候補: （親が選定した動画URL等。無ければ「なし」）
```

## 生成プロンプトテンプレート

以下をそのまま依頼文にする（[ ] を埋める）:

> Ablearn のコンテンツを生成して。docs/content-generation.md の手順に従うこと。
> 教科=[教科] 単元=[単元] 範囲=[範囲]。
> 範囲内の概念を列挙し、概念ごとに「概念ファミリー」を作って
> `public/content/chugaku/[教科dir]/` のセットJSONにまとめ、index.json に登録、
> `npm run validate:content` を通して結果を見せて。

## 概念ファミリーの構成

**1概念 = 確認したい知識1つ**（例:「同類項をまとめられる」「化学反応式の係数を合わせられる」）。
概念ごとに次を作る:

| 要素 | 内容 |
| --- | --- |
| 変種（stage0） | choice（4択）。`answers[]`（受理表記）を併記して input 形式でも出せるようにする |
| 変種（stage1） | input（自力入力）。同じ概念の別の数値・別の題材 |
| 変種（stage2） | `difficulty: 3` の応用（choice か input） |
| 自己説明 | choice・`difficulty` 1〜2。解き方の1ステップを見せて「なぜこの操作をした？」と理由を問う |
| 誤答修正 | choice・`difficulty` 1〜2。間違った解き方の例を見せて「どこが間違い？」と問う |
| 共通 | 同じ `concept` ID を付与（命名: `教科略-単元略-概念`。例: `sci-kagaku-keisuu`） |

- 自己説明・誤答修正は解き方に手順がある概念（数学の変形・理科の計算等）では必須。
  一問一答的な知識概念（用語・漢字等）では省略してよい
- 解き方に複数ステップがある概念には **faded example** を1問入れる:
  解答例の前半を見せ、後半を空欄にして埋めさせる（input か choice で表現。
  「【解答例のつづきを埋めよう】①…→②…→ x = ？」の形。
  専用形式は [25 写経](plans/25-trace-form.md) が入ったら活用）
- 各問題に `explanation` と `hints`（弱→強の2段階。
  1段目=考えるポイント、2段目=途中までの解き方）と `difficulty`（1/2/3）を付ける
- **`explanation` は「ステップ＋各ステップの理由」で書く**。結論だけの解説
  （「答えは○○だから」）は禁止。例: 「①y を移項する（x の項だけを残すため）→ 2x = 8 − y。
  ②両辺を2でわる → x = (8 − y)/2」。1ステップで答えが出る知識問題は理由1文でよい
- **別解がある問題は `explanation` で複数解法を比較する**（「別解: 先に両辺を4でわってもよい。
  移項を先にすると分数を扱う回数が減る」のように、どちらを選ぶかの観点まで添える）
- **熟達逆転効果の注意**: 自己説明・誤答修正・faded example は初学者向けの支援なので
  `difficulty` 1〜2 に置く。応用（`difficulty: 3`）には解答例的な過剰支援をつけない
  （hints 2段はそのまま。解答例を見せてから解かせる形式にしない）
- アプリ側の挙動: 同じ `concept` の変種は習熟度の段に合った**1問だけ**が出題され、
  `answers` つき choice は習熟が進むと自動で input 形式に変わる（だから受理表記の網羅が大事）
- 同じ段に変種が複数あるときは**ファイル順で先のものがラダー代表**になる。
  本変種（stage0〜2）を先に、自己説明・誤答修正・faded を後ろに並べる
  （後ろの変種は現状、模擬テストとレッスンで出題される）
- ID規約: `concept` は「確認したい知識」単位（1単元に複数あってよい）。セットIDは
  `教科略-ファイル名` に揃える。教科ディレクトリ名は `index.json` の既存構成
  （math / english / science / social / japanese / music / art / pe / tech-home）に従う
- 単元に1本、`kind: "lesson"` のレッスン（解説カード＋易しい問題のステップ列）を作る。
  順序は「説明 → すぐ確認」の交互。正解は本文で先に教えてから問う。
  解説カードの本文も「ステップごとの過程＋各ステップの理由」で書く（結論だけのカードは禁止）。
  **レッスン末尾に振り返りカード**を置き、「この単元で一番大事だったのは？」を
  自分の言葉で思い出させてから choice 1問で確認する
- 変種は**作成時に**作る（実行時生成はしない）

## 品質規約（READMEの注意に加えて）

- choice の誤答選択肢は「よくある間違い」から作る（符号ミス・係数の数え間違い等）。
  でたらめな選択肢は学習効果がない
- input の `answers` は漢字・ひらがな・別表記・順序違い（`2x+6y` と `6y+2x`）を網羅する
- order は「別の正しい並べ方」が存在しない文だけ（チャンク化 `"to play"` で一意にできる）
- **links は親が選定したものだけ**。生成時は「候補の提案」までに留め、
  index.json への登録は選定済みURLのみ
- 著作権: 教科書・市販ワーク・学校テストの問題をそのまま転載しない。
  同じ概念を確認する**オリジナルのアレンジ問題**として作る（公開リポジトリのため）

## 前提概念の宣言（concepts.json）

概念を新しく作ったら `public/content/<コレクション>/concepts.json` に宣言を追加する
（つまずき時の「前提概念への復習」誘導に使われる。[計画26](plans/26-prerequisite-backtrack.md)）:

```json
{ "id": "math-shiki-touhen", "name": "等式の変形", "set": "math-shiki-touhen",
  "prerequisites": ["shiki-doruiko", "math1-houteishiki"] }
```

- `id` は問題の `concept` と同じ。`set` はその概念のラダーが属するセットID
- `prerequisites` は「これが出来ないと解けない」直接の前提だけを書く（網羅より精度）
- 前提は **concepts.json 内で宣言済みのIDのみ** 参照できる。コレクション外
  （中1範囲など）の前提は `set` 無しのエントリとして宣言する（誘導には使われない）
- 循環参照は禁止（validate:content が検出する）

## 検証 → レビュー → 登録

1. `npm run validate:content` … スキーマ・id一意・answer範囲・形式チェック（必須）
2. `node scripts/validate-content.mjs --links` … リンクの死活（links を足したときだけ）
3. **Breaker レビュー**で内容の正確性（計算・事実・誤答選択肢の妥当性）を確認する。
   検証は形式までしか見ない（[計画14](plans/14-content-pipeline.md) の非スコープ）
4. `public/content/chugaku/<教科>/` に配置し、`index.json` の該当単元に登録して push
   （CIゲートが再検証する）

## 実例

- 新基準の実例: `public/content/chugaku/math/shiki-touhen.json`
  （等式の変形・1概念ファミリー。自己説明・誤答修正・faded example・別解比較・
  ステップ＋理由の explanation をすべて含む）。**形はこちらに合わせる**
- 旧基準の実例: `public/content/chugaku/science/kagaku-hannoushiki.json`
  （stage0〜2 の最小構成。遡及書き直しはしない方針 → [計画23](plans/23-content-quality.md)）
